## クイックリファレンス（毎回必ず確認）
1. 金額換算セクション必須（全記事）
2. キャラ対話2〜3箇所（オオタニ所長 + タナカ）
3. WPブロックは wp_block_builder.py 経由（手書きHTML禁止）
4. 価格・スペックはメーカー公式で必ず裏取り
5. 投稿後URLを必ず出力（`🔗 https://...`）
6. **画像は必ず image_resizer.py で2000px以下に**（APIエラー防止）
7. **JIN:R装飾ルール**：最重要強調は `***xxx***`（水色アンダーライン自動付与）。`**xxx**` に数値・日付・単位を含むと自動で水色アンダーライン
8. **前回記事のスタイル踏襲**：執筆前に直前記事のHTMLを取得し、装飾パターン・文体・画像配置を確認（MEMORY.md「前回記事スタイル踏襲チェック」参照）

### JIN:R装飾の使い分けガイド
| 書き方（Markdown） | 出力 | 用途 |
|---|---|---|
| `***キーワード***` | 水色下線付き太字 | 最重要：結論・パンチライン |
| `**54,780円**` | 水色下線付き太字（自動） | 数値・金額・日付・単位（自動検出） |
| `**ただの強調**` | 普通の太字 | 一般的な太字 |
| `*イタリック*` | 斜体 | 控えめな強調 |

**自動検出トリガー**（**xxx** 内にこれらがあると水色下線化）: `0-9` `０-９` `¥ 円 ％ % 日 分 時間 週 月 年 kg g mm cm km nit bpm ms 回 MB GB`

### 📚 曖昧復帰プロトコル（情報が不完全でも記事を特定できる）

過去記事を編集したいが、ファイル名・WP投稿ID・正確なタイトルを覚えていないケースへの対応。

#### 復帰の入口：ユーザーは「覚えている断片」だけ伝えればOK
```
「HUAWEIの記事を修正したい」
「ガジェットレビューで最近書いたやつ」
「ランナー向けのスマートウォッチの記事」
```

#### Claudeの動作フロー
```
1. article_status.py で曖昧検索
   python3 blog/scripts/article_status.py <キーワード>

2. 候補が1件 → 詳細取得 --detail で全情報表示
3. 候補が複数 → ユーザーに選択肢提示
4. 候補が0件 → 新規記事と判断、または別キーワードで再検索

5. 特定したら articles/xxx.md を Read（画像は読まない）
6. blog/MEMORY.md の記事台帳を参照（WP投稿ID・URL取得）
7. SKILL.mdルールに従って編集
```

#### 必須ツール
- `blog/scripts/article_status.py` — 曖昧検索・詳細表示・JSON出力対応
- 使い方：
  ```bash
  python3 blog/scripts/article_status.py              # 全記事一覧
  python3 blog/scripts/article_status.py huawei       # 部分一致検索
  python3 blog/scripts/article_status.py ガーミン     # 日本語OK
  python3 blog/scripts/article_status.py xxx --detail # 詳細+復帰コマンド
  python3 blog/scripts/article_status.py xxx --json   # 他スクリプト連携
  ```

#### 記事台帳（MEMORY.md）のメンテナンス
- 新規公開時：WP投稿ID・URL・公開日をblog/MEMORY.md「記事台帳」に追記
- **ID忘れても**：article_status.pyで検索可能。最悪WordPress管理画面から手動確認

### 🚨 画像の会話蓄積を防ぐ運用ルール（2000pxエラー根本対策）

**Readツールで画像ファイル（.png/.jpg）を直接読まない**。読むと会話履歴に埋め込まれ蓄積する。

#### 正しい画像の扱い方
| ケース | やり方 |
|---|---|
| 商品の外観を把握したい | サブエージェント（Explore/general-purpose）に分析を委譲し、**テキスト要約だけ**受け取る |
| 記事に画像を配置したい | パス文字列で指示（例：「blog/images/xxx.png をH2直後に」）。Claudeに画像を読ませない |
| どうしても確認が必要 | 1枚だけ Read → 確認後 `/compact` で履歴を圧縮 |
| 画像が大量にあるプロジェクト | **1記事＝1セッション**を徹底。記事終わったら新セッション |

#### 会話内の画像枚数が10枚を超えたら即 `/compact`
- Readで読んだ画像 + 添付画像 = 10枚で危険ゾーン
- `/compact` で履歴要約され画像が除外される（セッションは継続可）

#### サブエージェント活用パターン（推奨）
```
「blog/images/huawei/ 内の商品画像を分析して、
 記事に使える視覚的特徴をテキストで3つ挙げて。
 Explore エージェントに委譲してほしい」
```
→ 画像はサブエージェント内に閉じ、メインには文字情報だけ返る
- Claude API多数画像モードは**2000px以下必須**
- デフォルト上限1800px（200pxマージン）
- 長辺基準でアスペクト比保持

## 1. コンテンツジャンル

以下の6ジャンルを「効率化・費用対効果」の軸で企画する：
1. ガジェット ── 物理デバイス・ハードウェア全般
2. 時短術 ── 作業手順・習慣・思考法の改善
3. 時短ツール ── PCソフト・Webサービス・SaaS
4. AI ── ChatGPT・Claude・画像生成AI等
5. 効率化アプリ ── スマホ・PCアプリで業務・生活改善
6. 生産技術の裏話・工場勤務の実態 ── 現場のリアルな話

## 2. 記事構成テンプレート

### ガジェット・ツール・AI・アプリ記事
1. 引き込む導入文（冒頭3行で読者を離さない）
2. ツール・方法の紹介（名前・価格・概要）
3. リサーチ・口コミのサマリー（複数源から）
4. キャラクター対話を交えた解説
5. 生産技術視点での評価（4軸で）
6. 金額換算セクション（年間削減金額・回収期間を必ず記載）
7. 実際に使ってみた感想・数値
8. 深みのある考察（WHYの2段階・逆説・業界への問い）
9. 結論（どんな人に向くか・お得度の総括・一言締め）

### 裏話・工場実態記事
1. 引き込む導入（裏話型・共感型）
2. 具体的なエピソード（リアルさが命）
3. キャラクター対話
4. 「実はこれ、日常にも使える」という橋渡し
5. 関連するガジェット・ツールへの自然な流れ
6. 読者への問いかけで締める

## 3. ガジェット・ツール評価の4軸（必須）

1. **時短効果** ── 作業時間をどれだけ削れるか（分・時間で示す）
2. **効率化** ── 工程・手順をどう改善するか
3. **費用対効果** ── 金額換算で元が取れるか・いつ取れるか
4. **再現性** ── 誰でも同じ恩恵を受けられるか

---

## 4. 金額換算の公式（必ず盛り込む）

```
時給 × 削減できる時間（日） × 年間稼働日 = 年間削減金額
年間削減金額 ÷ 価格（または月額×12） = 回収期間
```

- 読者の時給は「2,000円」を標準として計算
- 数字は必ず3つセットで語る：価格・削減時間・年間換算金額
- 比較対象を明示する：「これを使わない場合、年間〇時間をムダにしている」

---

## 5. 導入文の型（冒頭3行で必ずどれかを使う）

- **問いかけ型**：「1日30分、何もしないのに消えている時間があるとしたら？」
- **数字衝撃型**：「年間125時間。それがこのツールを使わない人が失っている時間だ」
- **共感型**：「工場で働いていた頃、僕は毎日同じムダな作業を繰り返していた」
- **逆説型**：「効率化ツールを買いすぎると、逆に非効率になる」
- **裏話型**：「工場では絶対に教えてくれない、時短の本当の話をする」
- **口コミ型**：「Amazonに1,000件以上のレビューがある。その星1を全部読んだ」

## 6. キャラクター対話ルール

```
【タナカ】「〜〜〜（ボケ・勘違い・素朴な疑問）」
【オオタニ所長】「〜〜〜（ツッコミ＋深みのある一言）」
```

- 1記事2〜3箇所で挿入（難しい内容・裏話・金額換算の場面で効果的）
- タナカ：知識は浅いが素直で憎めない（急に賢くなるNG）
- オオタニ所長：生産技術の現場経験を持つ語り手（一人称は「僕」または「私」）
- **1セリフ＝1メッセージ**（詰め込みすぎNG）
- メタ表現禁止（「読者さんへの〜」など）
- **語調を絶対に変えない**（オオタニは断言口調、タナカは敬語口調）

## 7. リサーチルール

### 情報源リスト
- Amazon・楽天のレビュー（星1〜5をまんべんなく）
- X（旧Twitter）のリアルタイム口コミ
- Reddit・Quoraの海外ユーザーの声
- YouTubeのコメント欄
- メーカー公式サイト・スペックシート
- 専門メディアのレビュー記事
- Googleトレンド・Amazon売れ筋ランキング
- Makuake・Kickstarterの注目プロジェクト

### 星1レビューの活用
- 「どこが不満か」＝読者が本当に気にしているポイント
- 星1の不満を先に記事で取り上げ「でもこの使い方なら解決する」と返す構成が強い

### 口コミを記事に組み込む型
- 要約・言い換えで使う（直接引用・コピペ禁止）
- 「多くのユーザーが〇〇と感じている」など集合知として表現
- ネガティブ口コミは「こう使えば解決する」という提案につなげる
- 数字が出ている口コミは積極的に活用

## 8. SEOルール

### キーワード戦略
- メインキーワード1つ・サブキーワード2〜3つを記事作成前に設定
- メインキーワードはタイトル・最初の100文字・H2見出しに自然に含める
- 詰め込みはしない

### タイトル
- 32文字以内を目安
- 数字・金額・「裏話」「実態」「口コミ」で興味を引く
- 例：「【年間8万円の価値】〇〇レビュー｜生産技術視点で徹底検証」
- 例：「【1,000件の口コミを読んだ】〇〇レビュー｜星1から見えた本当の評価」

### メタディスクリプション
- 120文字前後。メインキーワードと記事の一番おいしい部分を含める

### E-E-A-T（経験・専門性・権威性・信頼性）
- 実際に使った経験・数値データを必ず入れる
- 複数の口コミ・情報源を統合した信頼性を示す
- 工場・生産技術の現場経験を根拠として示す

## 9. 文章校正チェックリスト

記事完成後、公開前に以下を必ずチェックする：

### 文章品質
- [ ] 文法・句読点の誤りがないか
- [ ] 一文が長すぎないか（50文字超は分割検討）
- [ ] 同じ意味の繰り返し・冗長表現がないか
- [ ] です/ます調とだ/である調が混在していないか
- [ ] 漢字とひらがなのバランスが適切か

### キャラクター品質
- [ ] オオタニ所長の断言口調が崩れていないか
- [ ] タナカの敬語口調が崩れていないか
- [ ] 1セリフ1メッセージになっているか
- [ ] メタ表現が入っていないか

### 金額・数値
- [ ] 「便利」「効率的」だけで金額換算していない箇所がないか
- [ ] 年間削減金額・回収期間が記載されているか

## 10. ファクトチェック手順

### 検証対象
- 製品名・モデル名・発売日・価格・スペック数値・機能名・技術名

### 情報源の優先順位
1. メーカー公式サイト（最優先）
2. 公式プレスリリース
3. 大手テックメディア（Impress Watch・PC Watch等）
4. Amazon・楽天の公式ストア

### よくある失敗パターン
| 失敗 | 対策 |
|---|---|
| 発売年を推測で書いた | メーカープレスリリースで必ず確認 |
| 価格を「〜万円台」と曖昧に記述 | 定価は税込・税抜を明記 |
| 複数モデルのスペックを混同 | 型番ごとに仕様を確認 |
| バッテリー持続を最良条件だけ記載 | 測定条件を明記 |

## 11. WordPress投稿ルール

- WordPress REST API（wp_api.py）で投稿
- **デフォルトは「下書き保存」**
- ユーザーの最終承認後に「公開」に変更
- カテゴリ・タグ・アイキャッチ画像を必ず設定

## 12. SNS投稿ルール（X）

- 140字以内
- 冒頭は金額・数字・「工場では〜」で引きつける
- ハッシュタグは末尾に3〜5個
- 推奨タグ：#ガジェット #時短 #効率化 #生産技術 #費用対効果 #AI活用

### SNS投稿の型
- 数字型：「月1,000円で年間25万円分の時間を生み出すツールを見つけた。」
- 裏話型：「工場では絶対に教えてくれない話をブログに書いた。」
- 口コミ型：「Amazonの星1レビューを100件読んで気づいたこと。」

---

## 13. アイキャッチ画像ルール

### スコアリング基準（実機写真が複数ある場合）
1. 人・手首着用あり → +3点
2. 画面点灯・表示あり → +2点
3. 正面アングル → +2点
4. 明るさ・ピントが良好 → +1点
5. 裏面・側面のみ → -1点

### 配置ルール
- 全画像を使わず「要所で効果的に」配置
- 配置の基本：スペック紹介直後・装着感セクション・耐久性レポート
- 同じアングルの写真が複数ある場合は最良の1枚だけ

---

## 14. 深みと説得力を出す技術

- WHYを2段階掘り下げる（表面→1段階→2段階）
- 逆説を使う：「安いガジェットほど高くつく理由」
- 現場の失敗談・成功談を入れる
- 結論を先に言い切る：「結論から言う。このツールは買いだ」
- 読者の反論を先に潰す：「高いと感じる人もいるかもしれない。でも…」

---

## 15. 承認フロー（3回チェック）

1. **企画レビュー** ── テーマ・キーワード・構成案をユーザーに確認
2. **記事レビュー** ── 完成記事の最終確認・修正
3. **改善レビュー** ── 公開後の分析結果の共有

---

## 16. 禁止事項

- 「便利です」「時短になります」だけで金額換算しない記述
- 根拠のない誇張（「神ツール」「最強」など）
- 口コミの直接引用・コピペ
- 現場経験のない薄い表現（「便利そうです」など）
- キーワードの不自然な詰め込み
- 振り返りレポートの省略
- キャラクターの語調変更
